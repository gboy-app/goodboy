# Security

Flow Engine is a credential router. Credentials flow through the process in memory, and the security model is organised around that fact: as little on disk as possible, explicit trust boundaries wherever processes talk, and every subprocess narrowed down to what it actually needs.

## Core properties

- **Zero persistence.** `SecuredBox`, the RAM-only workspace that holds credentials during a flow, never writes to disk. No IPC files, no temp files, no backup copies. Process exit clears it.
- **Keychain over files.** Operational parameters that must persist — browser Safe Storage keys, KeePassXC database passwords, Bitwarden session state — live in the macOS Keychain via `SecItem`. Access is gated by the user's login and the binary's code-signing identity.
- **AppDB is metadata only.** SQLite at `~/Library/Application Support/Goodboy/goodboy.db`, mode `0600`. Holds device configurations, settings, and logs. Never user credentials.

## Trust boundary

`goodboy-mcp` speaks MCP over stdio. Trust is established at configuration time: the user points their MCP client at the binary's absolute path, and anything that can spawn that binary is trusted to call any tool. There is no per-tool approval gate in `goodboy-mcp` — tool results surface every flow as a `FlowEvent` stream (the Glass Box principle), so the user's client sees each subprocess spawn and database read, but approval is post-hoc.

## Subprocess hardening

- `CLIRunner` builds a fixed `PATH` and a minimal environment allowlist. Parent environment is not inherited.
- Passwords pass over stdin where the CLI supports it (Bitwarden `--passwordfile /dev/stdin`). `op` and Proton Pass retain environment-variable passthrough with stderr redaction because their CLIs do not support stdin passwords.
- `keepassxc-cli` argv uses `--` sentinels; path arguments reject `..` and null bytes.
- Installed CLI binaries are verified against expected Apple Developer Team IDs (KeePassXC `G2S7P7J672`, Bitwarden `LTZ2PFU5D6`, 1Password `2BUA8C4S2C`). Mismatches are logged and surfaced as a soft warning in v1; hard rejection will follow once we have field data confirming no false positives.

## JSON Export

The `json` tool writes credentials to a single user-specified file. Hardening:

- Paths resolve (after tilde expansion, symlink resolution, and `standardizedFileURL`) into `~/Downloads`, `~/Documents`, or `~/Desktop`. Any other location is rejected.
- `..` path components and basenames beginning with `.` are rejected.
- The `.json` suffix is required.
- Output files are written with mode `0600`.

## Keychain parameter validation

`goodboy_keychain_set` validates every value before it reaches `SecItemAdd`:

- `serverUrl` must be an absolute `https://` URL with a host.
- `safeStorageKey` must base64-decode to exactly 16 bytes.
- `clientId` must match the RFC 4122 UUID shape.
- `serviceAccountToken` must start with `ops_` and be 32–256 characters.
- Values longer than 4 KB or containing null bytes are rejected for any key.

Rejection messages name the rule that failed; they never echo the rejected value.

## Pasteboard

Copied passwords are tagged with `org.nspasteboard.ConcealedType`, `org.nspasteboard.AutoGeneratedType`, and `com.apple.is-sensitive` (which blocks Universal Clipboard). The pasteboard auto-clears after 60 seconds if its contents haven't changed.

## Logging

`os.Logger` with `privacy: .private` on every URL, username, label, and credential-adjacent field. Identifiers, counts, and status codes stay `.public` so crash reports and Console output remain diagnosable.

## Known trade-offs shipping in v1

- **CLI Team ID mismatches are soft warnings, not rejections.** Hard rejection ships once dogfooding confirms no legitimate third-party forks trip the check.
- **Chrome vault authentication is source-level only.** Chrome's AES-256-GCM + PBKDF2 scheme protects the SQLite blob but does not authenticate the *origin* of the blob — standard for the Chromium vault family, and consistent with how Chrome itself treats the file.

## Reporting

Send security reports to `security@gboy.app`. Please do not open public issues for undisclosed vulnerabilities.
